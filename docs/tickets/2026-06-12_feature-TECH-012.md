## Epic: API Gateway

- [ ] `GatewayApplication.java` exists at `com.voidfuldawn.encountersofthevoid.gateway`
- [ ] Multi-stage Dockerfile uses Alpine Temurin images

**Resolution:** The gateway module was created with Spring Cloud Gateway routes using service-name URIs suitable for container-to-container networking. Actuator health endpoint exposure and port 8080 were configured in `application.yaml`.

## Epic: Code Quality

### [Technical] Fix Spring Cloud BOM version to 2024.0.x

**Story:** As a developer, I want the Spring Cloud BOM version to match the Spring Boot 3.3.x train, so that dependency resolution is consistent and conflict-free.

**Resolution:** The initial implementation used Spring Cloud BOM `2023.0.3`, which targets Spring Boot 3.1/3.2. The version was corrected to `2024.0.0` (aligned with Spring Boot 3.3.x) in the root `pom.xml`.

---

### [Technical] Correct package naming for all four domain modules

**Story:** As a developer, I want all domain module packages to follow the `com.voidfuldawn.encountersofthevoid.<domain>` convention, so that the codebase is consistent with the project specification.

**Resolution:** All four domain modules initially used compound suffixes (`userservice`, `layoutservice`, `campaignservice`, `templateservice`). Package declarations and directory trees were renamed to the required single-word suffixes: `user`, `layout`, `campaign`, `template`.

---

### [Technical] Move application-test.yaml to src/test/resources in all domain modules

**Story:** As a developer, I want test configuration files to live in `src/test/resources/`, so that they are excluded from the production JAR.

**Resolution:** `application-test.yaml` was incorrectly placed under `src/main/resources/` in all four domain modules, meaning it would have been bundled into each production artifact. The files were moved to `src/test/resources/` in every affected module.

---

### [Technical] Standardise prod datasource env-var names across all domain modules

**Story:** As a developer, I want `application-prod.yaml` to use `${DB_URL}`, `${DB_USERNAME}`, and `${DB_PASSWORD}` consistently, so that container orchestration environment variables are unambiguous.

**Resolution:** Initial implementations used non-standard variable names (`${DB_HOST}`, `${DB_PORT}`, `${DB_NAME_*}`, `${DB_USER}`, `${DB_PASS}`). All four `application-prod.yaml` files were updated to use the three canonical variable names specified.

---

### [Technical] Add spring.application.name to all service application.yaml files

**Story:** As a developer, I want each service to declare its application name, so that logs and service discovery are correctly labelled.

**Resolution:** `spring.application.name` was missing from all five `application.yaml` files. Each was updated with the appropriate name matching the module (e.g., `user-service`, `layout-service`, etc.).

---

### [Technical] Fix gateway route URIs to use service hostnames

**Story:** As a developer, I want the gateway to route to service hostnames rather than `localhost`, so that container-to-container networking functions correctly in Docker Compose and Kubernetes.

**Resolution:** All four gateway route `uri` values were changed from `http://localhost:808x` to the correct service-name URIs (`http://user-service:8081`, `http://layout-service:8082`, `http://campaign-service:8083`, `http://template-service:8084`).

---

### [Technical] Expose actuator health endpoint in gateway

**Story:** As a developer, I want the gateway's actuator health endpoint to be explicitly exposed, so that orchestration platforms can perform liveness and readiness checks.

**Resolution:** `management.endpoints.web.exposure.include: health` was added to `gateway/src/main/resources/application.yaml` as required by the specification.

---

### [Technical] Fix Dockerfiles: install Maven, copy root pom, and use -q flag

**Story:** As a developer, I want all module Dockerfiles to be buildable from their respective build contexts, so that CI/CD can produce container images without errors.

**Resolution:** Three issues were corrected across all five Dockerfiles: (1) `apk add --no-cache maven` was added to the builder stage because `eclipse-temurin:21-jdk-alpine` does not include Maven; (2) the root `pom.xml` is now copied into the builder stage so Maven can resolve the parent descriptor; (3) the Maven flag was changed from `-B` (batch mode) to `-q` (quiet mode) as required by the specification.

---

### [Technical] Add @SpringBootTest context-load tests to all domain modules

**Story:** As a developer, I want a full Spring Boot context-load test (`@SpringBootTest @ActiveProfiles("test")`) in each domain module, so that JPA, datasource, and security wiring is verified beyond the slice tested by `@WebMvcTest`.

**Resolution:** A dedicated context-load test class was added to each of the four domain modules alongside the existing `@WebMvcTest` slices. The `@ActiveProfiles("test")` annotation ensures the H2 in-memory datasource is used during context startup.

---

### [Technical] Add @ActiveProfiles("test") to GatewayApplicationTest

**Story:** As a developer, I want the gateway context-load test to activate the test profile, so that it does not accidentally pick up production or default configuration during CI.

**Resolution:** `@ActiveProfiles("test")` was added to `GatewayApplicationTest.java`, ensuring the test profile is explicitly activated when the gateway application context is loaded in tests.

---
